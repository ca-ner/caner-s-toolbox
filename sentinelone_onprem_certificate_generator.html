The Problem

The Common Name (CN) you entered is , but this hostname is not covered by the Subject Alternative Names (SAN) that will be generated from domain (i.e., ).

Why This Is Wrong (RFC 5280 / RFC 2818 / CA/Browser Forum)
  • Per RFC 2818 §3.1, TLS clients must use the SAN extension for hostname validation and ignore the CN field if SANs are present. If the CN is not in the SAN, the certificate will fail hostname verification.
  • Per CA/Browser Forum Baseline Requirements, any FQDN that a certificate is intended to cover must appear explicitly in the SAN extension — the CN alone is insufficient and deprecated for this purpose.
  • Modern browsers (Chrome 58+, Firefox, Edge) and TLS libraries (OpenSSL 1.1+) enforce SAN-only validation. A certificate whose CN is not in the SAN will produce a hostname mismatch error even if the CN looks correct.
  • SentinelOne Management and EDG require the server's FQDN to be resolvable and present in the SAN. A mismatch will cause agent connectivity failures and console errors.

  • Or, set the Common Name to a hostname that is covered by the SAN wildcards — either exactly matching the domain or a single-level subdomain of it.
